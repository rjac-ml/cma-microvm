## What this is

A reference solution that runs [Claude self-hosted sandbox](https://platform.claude.com/docs/en/managed-agents/self-hosted-sandboxes)
tool execution inside [AWS Lambda MicroVMs](https://docs.aws.amazon.com/lambda/).
Pattern: **orchestrator + ephemeral MicroVM per session**. Anthropic's control
plane handles orchestration; an event-driven control plane in the user's AWS
account launches one fresh, isolated MicroVM per Claude session. See `README.md`
for the full narrative and `docs/architecture.png`.

## Two independent codebases

This repo contains **two separate runtimes** that never share code. Understand
which one you are touching before editing:

1. **Launcher Lambda** (Python) — `src/functions/`. A SAM-packaged Lambda that
   verifies the inbound webhook and calls `RunMicrovm`. Built/deployed via SAM
   (see below). `pyproject.toml` declares it as a package rooted at
   `src/functions` (the `shared.*` import namespace).
2. **In-MicroVM worker** (Node.js) — `src/microvm-image/`. An HTTP lifecycle-hook
   server baked into the MicroVM image. Built with `npm` inside the Dockerfile
   and snapshotted at image-build time. Has its own `package.json`; do not mix
   its deps with the Python side.

The root `requirements.txt` / `pyproject.toml` are for the **operator-side**
scripts (`src/scripts/verify.py`) run from a developer's machine, not for the
Lambda runtime. The Lambda's runtime deps live in `src/functions/requirements.txt`
and are packaged by `sam build`.

## Build, deploy, and verify

These are AWS SAM + AWS CLI workflows. Deploy steps require AWS credentials and
an Anthropic agent/environment — they are not pure local builds.

```bash
# Control plane (launcher Lambda + REST API + WAF + secrets + roles):
sam build
sam deploy --guided --capabilities CAPABILITY_NAMED_IAM \
  --parameter-overrides "AnthropicEnvironmentId=env_..."

# Populate the two secrets the stack created (out-of-band, operator-only):
aws secretsmanager put-secret-value --secret-id <EnvironmentKeySecretArn>   --secret-string "<env-key>"
aws secretsmanager put-secret-value --secret-id <SigningSecretArn>          --secret-string "<whsec_...>"

# Build the MicroVM image (zip src/microvm-image/ → S3 → create-microvm-image):
./src/scripts/build-image.sh [stack-name]

# End-to-end verify from the operator's machine (uses the org API key, never AWS compute):
export ANTHROPIC_API_KEY="sk-ant-..." ANTHROPIC_ENVIRONMENT_ID="env_..." AGENT_ID="agent_..."
python src/scripts/verify.py --create
```

`samconfig.toml` and `.aws-sam/` are generated by SAM and git-ignored. There is
no test suite in the repo yet; `CONTRIBUTING.md` references `pytest` as the
intended tool for the `shared.*` modules (pure logic, no AWS needed).

## Architecture: the request flow

The control plane is **event-driven with no poller**; the only inbound traffic
is the webhook. The credential boundary is the most important invariant —
preserve it in any change:

1. A Claude session reaches running state → Anthropic POSTs a
   `session.status_run_started` **webhook** to API Gateway `/webhook`.
2. API Gateway request validation + WAF WebACL reject malformed/flooded
   traffic. **These are hygiene, not authentication.**
3. The **launcher Lambda** verifies the **webhook signature in-process**
   (`anthropic` SDK `client.beta.webhooks.unwrap`) using the signing secret from
   Secrets Manager. This HMAC check is the *only* authentication — there is no
   API Gateway authorizer (a REQUEST authorizer never receives the raw body
   needed to verify the HMAC). Failure → 401 before any MicroVM launches.
4. The launcher calls `RunMicrovm` with the session dispatch delivered via
   `runHookPayload`. It **dedupes on webhook event id** (DynamoDB-backed
   Powertools Idempotency, so retried/concurrent deliveries launch exactly one
   VM) and enforces the 5 TPS RunMicrovm rate limit (token bucket).
5. The **MicroVM** receives the dispatch on its `/run` hook, fetches the
   environment key from Secrets Manager using its own execution role, claims the
   matching session from Anthropic's work queue, runs the agent's tool calls in
   `/workspace`, posts results back, and exits; the idle policy then
   suspends/terminates the VM.

**Credential boundary (must not break):** The organization API key is
operator-only and **never** reaches AWS compute or a MicroVM. The launcher reads
**only the signing secret** and passes only a *reference* (secret ARN) to the
environment-key secret into the VM. The **MicroVM's execution role** reads
**only** the environment key. `src/functions/shared/payload.py` enforces this
with a `_FORBIDDEN_KEYS` guard — never put `ANTHROPIC_API_KEY` or
`ANTHROPIC_ENVIRONMENT_KEY` in the run-hook payload.

## Key files and why they exist

- `template.yaml` — the entire control plane: launcher Lambda + REST API, WAF
  WebACL, DynamoDB idempotency table, two Secrets Manager secrets, MicroVM
  execution role, and the S3 artifact bucket + build role used only for image
  builds. Comments in this file document subtle AWS quirks (the IAM action is
  `lambda:RunMicroVm` though the API is `RunMicrovm`; `lambda:PassNetworkConnector`
  for the AWS-managed connectors; `iam:PassRole` without a `PassedToService`
  condition; WAF log groups must be prefixed `aws-waf-logs-`). Read these
  comments before touching IAM/WAF.
- `src/functions/launcher.py` — webhook verification + launch orchestration.
  `Launcher` is pure logic (testable without AWS) and is decoupled from AWS via
  an injected `MicroVmClient` Protocol and a swappable `executor` (the executor
  is wrapped in DynamoDB idempotency only when `IDEMPOTENCY_TABLE` is set).
- `src/functions/shared/microvm_client.py` — `Boto3MicroVmClient` calling the
  `lambda-microvms` service through **vendored** boto3/botocore wheels in
  `src/functions/wheels/` (the Lambda runtime lacks this service model).
  `requirements.txt` references these wheels by file path.
- `src/functions/shared/rate_limiter.py` — `TokenBucket` enforcing the
  RunMicrovm 5 TPS limit.
- `src/microvm-image/worker/worker.mjs` — the in-VM hook server. Lifecycle hooks
  served on port 9000 under `/aws/lambda-microvms/runtime/v1/`; only `/run`
  does real work (ack-then-run, then `process.exit` so the idle policy cleans
  up). A `sessionStarted` guard ensures each VM handles at most one session.
- `src/scripts/build-image.sh` — zips `microvm-image/` with the Dockerfile at
  the archive root (required by `create-microvm-image`), uploads to the stack's
  artifact bucket, and creates the image with all lifecycle hooks enabled.

## Conventions

- The launcher is written to run on **Python 3.14 / arm64** (see `template.yaml`
  `Globals`). `anthropic[webhooks]` + `aws-lambda-powertools` are imported at
  module scope so cold-start cost is paid once.
- Malformed webhook events return **200** (not 4xx) so Anthropic does not retry
  — a malformed event will never become valid. Only launch failures and
  signature failures return non-2xx.
- This repo uses **Spec-Driven Development (speckit)** — the
  `/speckit-*` skills drive specify → plan → tasks → implement. The SPECKIT
  block above is where speckit injects the current plan; keep it intact.
- Per the user's standing preferences: for Python, prefer `uv add` / `uv init`
  with `pydantic-settings` and `loguru`; do not add a `version:` key to
  docker-compose files; do not run `make` targets — ask the user to run them.
